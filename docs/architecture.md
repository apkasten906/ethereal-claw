# Architecture

`ethereal-claw` keeps the orchestration engine independent from the user interface.

Related docs:

- [Command reference](command-reference.md)
- [Workflow](workflow.md)
- [Prompt docs](prompts/ideation.md)

## Flow

1. The CLI parses a command and validates input.
2. The core orchestrator creates or reuses a feature workspace.
3. Agents produce artifacts through a provider abstraction.
4. Budget and token tracking are updated per workflow step.
5. Each run writes a stable JSON log under `ec/runs/` by default.

## Packages

- `packages/cli`: CLI surface, command registration, console presentation.
- `packages/core`: workflow orchestration, agents, providers, artifacts, logging, config, git, budget.
- `packages/shared`: workflow stages and portable domain types.

## Runtime Artifacts

Runtime artifacts are intentionally file-based so every stage can be inspected and reviewed.

- `ec/features/<feature-slug>/feature.yaml`: stable metadata for a feature workspace.
- `ec/features/<feature-slug>/**`: stage artifacts generated by commands.
- `ec/features/<feature-slug>/run-history/*.json`: run logs scoped to one feature.
- `ec/runs/*.json`: global run log copies.
- `config/ethereal-claw.config.yaml`: local provider, artifact root, and budget configuration.

`baseDirectory` in `config/ethereal-claw.config.yaml` controls the artifact root. The default is `ec`.

See the [artifact map](command-reference.md#artifact-map) for command-level ownership.

## Command Boundary

The CLI commands are thin wrappers around the core orchestrator:

- `init` prepares local runtime directories and config.
- `ideate` creates a feature workspace.
- `plan`, `implement`, `test`, and `review` operate on an existing feature workspace.
- `run` chains `plan`, `implement`, `test`, and `review`.

The scaffold currently produces artifacts and logs. It does not autonomously edit application source files, run the repository test suite, submit PR reviews, merge branches, or deploy code.
